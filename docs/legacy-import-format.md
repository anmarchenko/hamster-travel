# Legacy Import Format

This document defines the conversion output generated by:

- `uv run python scripts/convert_legacy_csv_for_import.py`

## Input

- Cleaned legacy CSV set (default): `prod_backup/legacy_csv_clean/`

## Output

- Output directory (default): `prod_backup/import_ready/`
- Files:
  - `trips.jsonl`
  - `warnings.jsonl`
  - `summary.json`
  - `legacy_user_mapping.template.json`
  - `day_index_map.csv`

## File Contracts

### `trips.jsonl`

One JSON object per trip bundle.

Top-level fields:
- `trip_ref` (`legacy_trip_<id>`)
- `legacy_trip_id`
- `name`, `status`, `dates_unknown`, `start_date`, `end_date`, `duration`
- `currency`, `people_count`, `private`
- `author_legacy_user_id`
- `participant_legacy_user_ids`
- `destinations`
- `activities`
- `accommodations`
- `transfers`
- `day_expenses`
- `food_expense` (nullable)
- `notes`
- `source_counts`

Key mapping decisions:
- User references stay as legacy IDs, to be mapped at import time.
- Geo references use geonames IDs (`city_geonames_id`).
- Legacy `days` are converted to `day_index` (0-based per trip).
- Multiple cities on the same day are preserved via overlapping destination ranges.
- Legacy plain-text descriptions/comments are converted to rich HTML expected by the current editor/rendering pipeline:
  - paragraphs: `<p>...</p>`
  - line breaks inside paragraph: `<br>`
  - links: `<a href=\"...\" target=\"_blank\" rel=\"noopener noreferrer\">...</a>`
- Legacy trip short description is not imported.
- Legacy trip comment is imported as a note with title `–û—Ç—á–µ—Ç –æ –ø—É—Ç–µ—à–µ—Å—Ç–≤–∏–∏`.
- Legacy food descriptions are imported as a trip-level note with title `–ï–¥–∞`.

### `warnings.jsonl`

One warning per line for lossy/fallback conversions, for example:
- unknown transfer mode fallback
- missing transfer times fallback
- missing city mapping causing transfer skip
- trip has invite rows (not imported)

### `summary.json`

Conversion metadata:
- `format_version`
- `created_at_utc`
- source/output paths
- legacy user IDs found
- trip/warning counts
- notes for importer behavior

### `legacy_user_mapping.template.json`

Template map of legacy user IDs to target app users; values are `null` placeholders.

Example:
```json
{
  "191": null,
  "192": null
}
```

### `day_index_map.csv`

Debug/reference mapping:
- `trip_ref`
- `legacy_trip_id`
- `legacy_day_id`
- `day_index`
- `date_when`
- `legacy_index`

## Import Command

Use the mix task:

```bash
mix legacy.import_trips \
  --bundle-dir prod_backup/import_ready \
  --user-map-file /absolute/path/to/legacy_user_mapping.json
```

Optional flags:
- `--limit N`
- `--skip-missing-cities`
- `--continue-on-error`

Example mapping file (`legacy_user_mapping.json`):

```json
{
  "191": "author@example.com",
  "192": "participant@example.com"
}
```

Mapping values can be either user email or user UUID in the target environment.

Important: importer run starts by deleting existing trip-related data (clean-state import).
